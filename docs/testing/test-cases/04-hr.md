# 04 — HR & Payroll (`/hr`) test cases

UI test cases for the M2 HR & Payroll module: employees, org structure, OT approvals, cash-advance
approvals, attendance, payroll runs, and tax exports. High-risk focus: payroll calculate→approve,
cash-advance Super-Admin re-auth, salary masking (`hr.salary.view`), and OT submit/approve.

**Coverage checklist**

- [x] Smoke: module loads, employees list renders (TC-HR-01)
- [x] Golden path: create employee → detail → edit → salary record (TC-HR-02)
- [x] Permission gate: view-only persona, disabled-with-tooltip create actions (TC-HR-03)
- [x] High-risk: salary masking everywhere `MaskedValue` is used (TC-HR-04)
- [x] Org structure: departments/positions, department-first flow (TC-HR-05)
- [x] OT: create drawer validation + create→submit (TC-HR-06), approve queue (TC-HR-07)
- [x] Cash advance: approve with re-auth (TC-HR-08), reject with reason (TC-HR-09)
- [x] Attendance: month grid + import (TC-HR-10)
- [x] Payroll: runs list + create (TC-HR-11), wizard calculate→approve (TC-HR-12), breakdown drawer + review masking (TC-HR-13), approve permission gate (TC-HR-14)
- [x] Tax exports PND1/SSO job toasts (TC-HR-15)

> **Persona setup (applies to every non-super-admin case):** in the running app,
> `VITE_DEV_PERMISSIONS` is **not** honored — it only shapes the dev user in unit tests. A limited
> persona must be a **real user**: as `superadmin`, create a role with exactly the case's permission
> CSV (`/admin/roles` → Create role → tick codes in the matrix), create a user with that role
> (`/admin/users` → Create user, set a temp password), then sign in as that user.
> **Flagged test-data gap:** the seed provides only `superadmin` — there is no seeded "HR clerk" /
> "Payroll approver" persona or seeded employees/OT/advances; each suite run must self-provision.

### TC-HR-08 — High-risk: cash-advance approve requires Super-Admin re-auth
- **Target**: App @ :5173 `/hr/advances`
- **Persona**: super-admin, plus a non-super-admin HR persona for the negative half
- **Preconditions**: a SUBMITTED cash advance exists. **Flag (gap):** there is no create-cash-advance UI — seed via API `POST /cash-advances` (contract `createCashAdvance`) before the test.
- **Priority**: high-risk
- **Steps**:
  1. As the non-super-admin persona, open `/hr/advances`; hover the "Approve" button on a card.
  2. As `superadmin`, open `/hr/advances`. Verify the card: employee name, "Amount" (money), "Reason" (or "No reason given"), and the ceiling badge.
  3. Click "Approve". In the dialog "Approve cash advance for {name}?", read the consequence text, type the super-admin password into "Super-Admin password" (placeholder "Re-enter to authorize"), click "Approve".
- **Expected**:
  1. Non-super-admin: "Approve" is disabled with tooltip "Requires Super-Admin"; "Reject" remains enabled.
  2. Ceiling badge shows exactly one of "Within ceiling" (✓, amount ≤ 50% of ceiling), "Approaching ceiling" (⚠), or "Over ceiling" (⚠, > 100%) — icon + label, never color alone.
  3. Dialog consequence: "This disburses the cash advance to {name} once processed."; confirm disabled until password entered.
  4. Toast "Cash advance approved."; card leaves the queue (filters `status=SUBMITTED`).
- **Automation notes**: **Flag (gap):** the collected password is *not sent* — `approveCashAdvance` contract body is `z.void()` and the page calls it with `body: undefined`; re-auth is enforced only as UI + backend super-admin role check. A wrong password therefore still succeeds — do not assert wrong-password rejection here. **Flag (gap):** no *disburse* UI exists (the `useDisburseCashAdvanceMutation` hook and `POST /cash-advances/:id/disburse` endpoint exist but no screen calls them) — disburse can only be covered at API level.

### TC-HR-12 — High-risk: payroll wizard — inputs → calculate → review → approve
- **Target**: App @ :5173 `/hr/payroll/runs/$id` (payroll-wizard)
- **Persona**: super-admin (holds `hr.payroll.approve` + `hr.salary.view`)
- **Preconditions**: a DRAFT run (TC-HR-11); ≥2 ACTIVE employees, one **without** a salary record (to trigger the flag); optionally an APPROVED-unreconciled OT request
- **Priority**: high-risk
- **Steps**:
  1. Open the run. Verify the step header (list "Payroll run steps"): Inputs · Calculate · Review · Approve; "Inputs" is current; period heading + status chip (draft).
  2. On Inputs: verify the warning banner "Resolve or exclude the flagged employees before calculating." and the flagged employee's badges ("Missing salary" danger, "Unreconciled OT" warning); confirm "Continue to calculate" is disabled.
  3. Tick the flagged employee's "Exclude" checkbox (aria-label "Exclude {name}").
  4. Banner flips to "No missing data — ready to calculate."; click "Continue to calculate".
  5. Click "Run calculation". Observe "Calculating…" + skeleton rows; wait for payslips.
  6. Click "Continue to review". Verify the review table (Employee / Net), Net values as money, any "Outlier" chip (net ≤ 0 or > 2× base).
  7. Click "Continue to approve". Verify the summary "N payslip(s) · Net total <sum>".
  8. Click "Approve payroll". In the dialog "Approve payroll run {period}?" read the consequence ("This locks payroll run …, pulls outstanding advances into deductions, and generates payslips."), click "Approve payroll".
  9. Reload the page.
- **Expected**:
  1. Steps ahead of the current one are disabled (not reachable); completed steps get a check.
  2. Calculate is blocked until every flagged employee is excluded or fixed.
  3. `POST /payroll-runs/:id/calculate` fires; payslips arrive by polling `GET /payroll-runs/:id/payslips`; run chip becomes in-progress (CALCULATED).
  4. Approve dialog needs **no** password and **no** reason (preset `payroll-approve`); confirm label "Approve payroll".
  5. Toast "Payroll run approved."; run chip becomes approved.
  6. After reload the wizard resumes at the step derived from status (CALCULATED→Review, APPROVED→Approve) — no lost state.
- **Automation notes**: no GET-single-run endpoint — status comes from the run list matched by id; assert via the chip, not network. Steps are `button`s inside `ol[aria-label="Payroll run steps"]` with `aria-current="step"`. Flag: calculation completion has no explicit UI event — poll for review-table rows.

**Flagged gaps (HR)**

1. No seeded limited personas or HR fixture data — every non-super-admin case must self-provision users/roles via the Admin UI, and employees/OT/advances via the UI or API.
2. OT "Approve" (row action + drawer button) is not client-side gated by `hr.ot.approve`; OT has no reject endpoint/UI.
3. Cash-advance re-auth password is collected but not transmitted (`approveCashAdvance` body is void) — wrong-password behavior cannot be asserted; there is no disburse UI and no create-advance UI.
4. Tax exports have no job-status endpoint — "file ready" is untestable from the UI.
5. Zero `data-testid`s across HR screens; ceiling badge, masked values, and wizard steps are reachable via ARIA/text only. Recommended additions: test ids on `MaskedValue`, `CeilingCheckBadge`, and DataTable row-action triggers.
6. Attendance import file format is undocumented in-UI; a committed fixture file is needed for repeatable runs.
